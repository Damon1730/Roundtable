## 二、Claude 代码审查结论

V0.4 核心链路扎实，但整体是按「单机单用户」假设编写。一旦加账号体系，以下结构性问题会从「无害」变成「安全/隐私 bug」。

### 必须在 V0.5 之前处理的地基问题

1. **缺少「用户」维度。** `meetings` 表只有 id/title/status/created_at，无 `user_id`。账号体系第一步是给数据模型加用户外键，这才是 V0.5 真正的工作量来源。

2. **WebSocket 与 REST 全局可见 / 全局广播。**
   - `GET /api/meetings` 返回所有人的会议列表。
   - HTTP 追问走 `broadcastJson` 推给每一个连着的客户端，前端仅靠 `meetingId` 过滤，数据其实已发到所有人浏览器 —— 多用户后即越权读取。
   - 必须把 WS 连接绑定登录会话，只推送给会议归属者。

3. **会议 ID 客户端生成且无归属校验。** `meeting-${Date.now()}` 可枚举；`GET /DELETE /api/meeting/:id` 都不校验归属，换个 ID 就能读/删他人会议。需改为服务端生成 + 归属校验。

4. **Token 用量已拿到但未落库。** connector 里有 `usage:{input_tokens,output_tokens}` 却被丢弃。VIP 分级需要按会议数/token 计量，这是「能分级」的计量基础，否则裸奔烧 DeepSeek。

> 建议：将以上四点打包为 **V0.45「多租户地基」** 小迭代先行（user_id + 服务端会议 ID/归属校验 + WS 按会话定向 + usage 落表），再上登录。

### 现存真实 Bug 与代码异味

- **追问会被永久卡死（真 Bug）：** 追问轮次靠 `messageCount % roles.length === 0` 反推。但系统容忍单角色失败，一旦某角色第一轮没发言成功，消息数除不尽，该会议追问永久返回 `meeting_not_ready`。根因是「消息总数 ÷ 角色数」推算轮次 —— 应给 messages 加 `round` 列显式记录。
- **发言归属靠正则反解 prompt：** `currentRoleName` 用 `以(.+?)身份发言` 从 prompt 文本抠角色名，改一句 prompt 措辞就可能错配。DAG 节点本就知道角色身份，应让 connector 从 config 直接拿到。

## 四、Claude 对各版本的具体建议

### V0.5（账号体系 + VIP）

- 路线已确认 Web SaaS，建议正式封存小程序计划，避免两头牵扯。
- **VIP 用配额而非功能阉割来分级**（阉割会让免费用户觉得产品残缺）：

  | 维度 | 免费 | VIP |
  |---|---|---|
  | 每日会议数 | 限 N 场 | 放开/高额度 |
  | 角色数上限 | ≤5 | ≤10 |
  | 追问轮次 | 1 轮 | 2 轮 |
  | 模型 | deepseek-chat | 可选 reasoner |
  | 导出 | 纯文本/MD | 全格式 |
  | 历史保留 | 30 天 | 永久 |

- **邀请系统：** `invite_codes` 表 + 兑换记录，邀请码可发「试用 VIP N 天」，兼作增长杠杆。
- **安全要点：** 密码用 argon2/bcrypt 哈希；session token（httpOnly cookie）或 JWT；WS 握手校验同一凭证。这是网络暴露的认证面，不自行手搓加密。

### V0.6（导出会议记录）

- 比 V0.5 简单、风险低（数据在 SQLite，内容本就是 Markdown）。
- 建议把 **Markdown 导出**提前/并行做（不依赖认证面，可独立快速上线，给演示加分）；PDF/Word 才需引库（`puppeteer` 转 PDF / `docx`）。
- **前提坑：** 当前 round/stage/总结分段全是前端用消息顺序临时推算（`decorateDiscussionMessages`、`buildSummarySections`）。要导出干净记录，需先把 `round`/`stage` 持久化到 messages 表 —— 与「消息协议结构化」是同一件事的两面，也正好修掉追问卡死 Bug。

### Claude 额外想法（按性价比排序）

1. 修追问卡死 Bug（合并进消息结构化）。
2. 发言归属改为从 config 取，不再正则反解 prompt。
3. **分享链接（只读公开页）** `/share/:token`：既是增长入口，又是邀请裂变载体，可与导出复用渲染。
4. **演示兜底模式：** 缓存 2-3 个高质量会议，现场 API 抖动时兜底（Codex 纪要也提过）。

## 八、阶段一实施记录：消息结构化 + 修追问卡死 Bug（2026-05-30 当日落地）

许先生拍板：先推进「消息结构化」。Claude 进入规划模式核查 agency-orchestrator 源码后实施，分两阶段，本节为**阶段一**成果。

### 关键技术核查结论（决定方案的两个发现）

1. **executor 调用 connector.chat 时只传 LLM 配置、不传 step 身份**（executor.js:251）——这正是旧代码被迫用正则 `以(.+?)身份发言` 反解角色名的根因。
2. **onStepComplete 是整批并发完成后才逐个触发**（executor.js:80-129），不是谁先返回谁先触发。因此发言落库/emit 必须留在 connector.chat 内，否则快角色发言会被同批慢角色拖到一起冒泡，破坏「快的先出、慢的不阻塞」的圆桌体验。→ 排除了「在 onStepComplete 里落库」这个看似更优雅但有害的方案。

### 实际改动（仅动 2 个文件，未碰前端呈现层，与 Codex 头像工作不冲突）

**`server/index.js`（净增约 138 行）：**
- messages 表新增 `stage` / `round` / `role_id` 三列；沿用既有 PRAGMA 迁移模式，旧库自动 ALTER 补列，向后兼容。
- 发言落库写入真实 `stage`（discussion / followup / summary）、`round`、`role_id`。role_id 现为可靠角色标识（如 `legal-legal-contract-reviewer`），**不再正则反解 prompt**，消除「改 prompt 措辞即错配」风险。
- 新增 `evaluateFollowupReadiness()`：追问就绪/轮次以 DB 为唯一真相（discussion 阶段有发言即 ready；nextRound = `stage='followup'` 的 MAX(round)+1）；服务端不再信任前端传的 `round`。
- 清除三处因重构产生的死代码（`normalizeFollowupRound`、`INITIAL_DISCUSSION_ROUNDS` 等）。

**`test-e2e.js`（+22 行）：**
- 新增针对卡死 Bug 的回归断言：构造「某角色第一轮超时失败」的会议，验证其仍能成功追问（旧逻辑此处永久卡死）。

### 修复的真 Bug

追问轮次原靠 `messageCount % roles.length === 0` 反推。系统容忍单角色失败 → 一旦某角色第一轮没发言成功，消息数除不尽 → 该会议追问永久返回 `meeting_not_ready`。已由 DB 权威轮次判断根治。

- Mock E2E：**112 项断言全过**（原 110 + 新增 2 项回归）。
- 直接查测试库确认：stage/round/role_id 三列正确落库，summary 的 round 为 null，followup 轮次计算准确。
