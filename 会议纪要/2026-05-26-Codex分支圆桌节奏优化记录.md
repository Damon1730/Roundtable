# Codex 分支圆桌节奏优化记录

**日期：** 2026-05-26
**分支：** `Codex`
**目标：** 让 Alpha 版从“多角色快速出报告”更接近“多角色正在开会聊天”的体感，便于后续 Beta 版手机演示和投资人体验。

---

## 背景判断

Alpha 版的核心能力已经成立：用户输入主题，系统自动选角，后端通过 `agency-orchestrator` 构建 DAG，并发调用 DeepSeek，最后通过 WebSocket 推送结果。

但体验上存在一个明显问题：角色回复太快、太整段、太像批量生成报告。根因不是模型能力不足，而是当前实现把“后台生成速度”和“前台会议节奏”绑定在一起：

1. 后端 `executeDAG` 并发执行角色 step，默认 `concurrency: 3`。
2. 每个角色生成完成后，服务端一次性推送完整 `speech`。
3. 前端收到 `speech` 后立即整段渲染。
4. 第一轮角色之间没有真正回应关系，总结也更像把多份报告合并。

因此这次优化的方向不是让模型更慢，而是把“计算”和“表演/阅读节奏”拆开：后台仍可快速并行生成，前台按会议节奏逐条播放。

---

## 技术实现思路

### 1. 保留现有技术底座

继续使用当前 Alpha 已跑通的架构：

- React + Vite 前端
- Node.js + WebSocket + SQLite 后端
- `agency-orchestrator` DAG 编排
- DeepSeek 作为 LLM connector
- `agency-agents-zh` 角色库

这避免在投资人演示前引入 Taro、小程序、登录、域名、审核等新风险。

### 2. 后端改成分阶段圆桌

原始流程：

```text
角色并行发言 -> 总结
```

现在改为：

```text
第一轮短发言 -> 第二轮基于第一轮互相回应 -> 会议总结
```

具体调整：

- 第一轮角色 prompt 限制为 180-320 字，要求像真实会议短发言，不写长报告。
- 第二轮从数据库读取第一轮已成功发言，生成一段 transcript，再让每个成功发言过的角色做 100-180 字回应。
- 总结阶段从数据库读取所有成功发言生成总结，而不是依赖静态 DAG 变量。
- 单个角色失败或超时不会阻塞整场会议；总结会基于已经完成的发言继续生成。
- 新增 `ROUNDTABLE_CONCURRENCY` 环境变量，默认仍为 3，便于后续按演示效果调节并发。
- 新增 `ROUNDTABLE_DISCUSSION_DELAY_MS` 环境变量，默认 900ms，给前端播放节奏使用。

### 3. 前端增加播放队列

前端不再收到 `speech` 后立即整段展示，而是：

```text
收到完整 speech -> 放入播放队列 -> 显示该角色正在思考 -> 延迟 -> 打字机逐段输出
```

具体调整：

- 新增播放队列，保证多条发言按顺序展示。
- 新增打字机输出，每次展示少量字符。
- 新增输出中的光标动画。
- 新增消息去重，避免 WebSocket 重连或重复事件导致重复播放。
- 服务端已经 `done` 但前端仍在播放时，按钮继续保持进行中，避免用户误触打断演示。

### 4. 测试同步更新

原 E2E 假设每个角色只产出一条 `speech`，现在初始会议变为两轮，所以测试同步调整为：

- 3 个角色应产出 6 条角色发言。
- 5 个角色应产出 10 条角色发言。
- 10 个角色应产出 20 条角色发言。
- thinking 与 speech 不再要求严格相邻，只要求每条 speech 前出现过对应角色的 thinking。
- Windows 下测试结束时不再强制 `process.exit()`，改为设置 `process.exitCode`，避免 WebSocket 句柄关闭时触发 Node 原生断言。

## 具体修改文件

| 文件 | 变更 |
|---|---|
| `server/index.js` | 分阶段圆桌流程、第二轮回应、总结基于已成功发言生成、追问轮次兼容初始两轮讨论 |
| `src/main.jsx` | 发言播放队列、思考延迟、打字机输出、去重、播放完成前保持运行态 |
| `src/styles.css` | 输出光标动画 |
| `test-e2e.js` | 更新两轮圆桌行为断言，修复 Windows 测试退出问题 |

## 后续建议

1. 手机演示版优先继续沿 H5 方向打磨，不急着迁移小程序。
2. 后续可以把“第二轮回应”做成更明确的主持人串场，例如“主持人点名某角色回应刚才的观点”。
3. 可以加演示模式：提前缓存 2-3 个高质量会议案例，现场网络或 API 波动时可兜底。
4. 投资人演示前建议控制角色数为 4-6 个，过多角色会拉长等待和播放时间。
5. 后续 Beta 若要正式上小程序，再处理微信登录、wss 合法域名、体验版成员、内容安全和审核类目。
